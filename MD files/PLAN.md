# الخطة الكاملة — منصة SaaS متعددة التجار (Multi-Vendor) لإدارة المخزون والمتجر العام

## 1. الفكرة
منصة تجارة إلكترونية زي أمازون: عدة تجار (vendors) بيبيعوا منتجاتهم عبر واجهة متجر عامة
واحدة، وكل تاجر بيدير مخزونه من داشبورد خاص فيه، والزبون **بيسجل حساب حقيقي** ليشتري —
تماماً متل أي موقع تجارة إلكترونية حقيقي. لا Shopify، لا زوار مجهولين، لا سلة بالمتصفح —
**كل شي محمي بحساب حقيقي و RLS**.

## 2. الأطراف الثلاثة — كلهم حسابات حقيقية موثّقة
| الطرف | شو بيعمل | كيف محمي |
|---|---|---|
| **التاجر (Vendor)** | بيدير متجره: منتجات، مخزون، طلبات، موردين، اشتراك | `org_id` عبر `memberships` |
| **أدمن المنصة (Platform Admin)** | يشوف كل التجار، يعدّل خططهم | جدول `platform_admins` منفصل |
| **الزبون (Customer)** | يتصفح بحرية، **لازم يسجل حساب** ليضيف للسلة أو يشتري | `customer_id = auth.uid()` |

**الاستثناء الوحيد:** تصفح الكتالوج (المنتجات المنشورة) عام وبدون تسجيل دخول — زي أي متجر
حقيقي. أي شي بعد هيك (سلة، شراء، ملف شخصي) لازم حساب.

## 3. البنية المعمارية
أي عملية كتابة أو قراءة لازم تمر عبر: **Server Action → Zod validation → Service Layer
(`lib/services/*.ts`) → Supabase client + RLS**. القاعدة الجديدة الصارمة: **ما في ولا
عملية كتابة واحدة بالمشروع بدون هوية موثّقة (`auth.uid()`)** — لا سلة زائر، لا طلب زائر.

## 4. تصميم قاعدة البيانات (محدّث بالكامل)

```
organizations (التجار)
├── id, name, slug, plan (free|pro), is_active, created_at

memberships (موظفين التاجر)
├── id, org_id, user_id, role (owner|admin|staff)

platform_admins
├── id, user_id, created_at

customers (حسابات الزبائن — جديد، منفصل عن memberships)
├── id (= auth.uid()), name, email, created_at

carts (جديد — سلة محمية بالداتابيز)
├── id, customer_id, created_at

cart_items (جديد)
├── id, cart_id, product_id, org_id, quantity

warehouses / product_stock / stock_movements / suppliers /
purchase_orders / purchase_order_items / payments / audit_log   ← كما هي

products (+ price, description, image_url, is_published)

orders (مربوطة بحساب الزبون الحقيقي، مش إيميل مكتوب يدوياً)
├── id, org_id, customer_id, shipping_address, status, total_amount, created_at

order_items
├── id, order_id, product_id, quantity, unit_price
```

### RLS — أربع فئات واضحة
1. **جداول التاجر**: `org_id` عبر `memberships` + `auth.uid()`
2. **جداول الزبون** (`customers`, `carts`, `cart_items`, `orders` كمشتري): `customer_id
   = auth.uid()` — الزبون يشوف ويعدّل بس بياناته هو
3. **الاستثناء الوحيد للقراءة العامة**: `products` حيث `is_published = true`
4. **ولا استثناء واحد للكتابة العامة** — كل INSERT/UPDATE لازم `auth.uid()` حقيقي

## 6. ملخص القرار السريع
| السؤال | الجواب |
|---|---|
| هل الزبون لازم حساب؟ | إي، إجباري — بدون حساب ما فيه يضيف للسلة أو يشتري |
| وين السلة مخزّنة؟ | بالداتابيز (`carts`/`cart_items`)، محمية بـRLS مربوطة بحساب الزبون |
| مين بيقرر الـ org_id وقت الشراء؟ | السيرفر، من المنتج نفسه — أبداً من طلب الزبون |
| هل في أي كتابة بالداتابيز بدون تسجيل دخول؟ | لأ، صفر — القاعدة الصارمة الجديدة بكل المشروع |
| مين بيقدر يعدّل خطة أي تاجر؟ | فقط Platform Admin أو PayTabs webhook الموثّق |
| وين MCP؟ | مرحلة 27 فصاعداً |
